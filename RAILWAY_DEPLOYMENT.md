# OXOT Website — Railway Deployment Guide

This document describes the production deployment infrastructure for OXOT-Website on Railway. The setup consists of two main services: **PostgreSQL** (database) and **OXOT-Website-JULY2026** (Next.js app).

## Architecture

```
┌─────────────────────────────────┐
│  OXOT-Website-JULY2026          │  (Next.js + Node)
│  - Dockerfile-based             │  - Handles preDeployCommand
│  - Port 3000                    │  - Runs migrations & seeds
│  - Auto-healthcheck (/en)       │  - Serves public website
└──────────────┬──────────────────┘
               │ DATABASE_URL
               ↓
┌─────────────────────────────────┐
│  PostgreSQL (pgvector)          │  (Database)
│  - Railway PostgreSQL template  │  - pgvector for embeddings
│  - 1GB volume (configurable)    │  - Auto-initialized schema
│  - Version: 15+                 │  - Persistent data
└─────────────────────────────────┘
```

## Prerequisites

- A Railway project with the OXOT-Website-JULY2026 service already created
- Git access to push changes (railway.json is part of the repo)

## Setup Steps

### Step 1: Merge the Dockerfile Fix (PR #6)

The build currently fails because `ENV NODE_ENV=production` is set before `npm ci`, skipping devDependencies. 

**Action:** Merge [PR #6](https://github.com/Planet9V/OXOT-Website-JULY2026/pull/6) on GitHub.

```bash
# Verify the fix is in main
git log --oneline | head -1
# Should show: "fix: move NODE_ENV=production after npm ci in Dockerfile"
```

### Step 2: Deploy PostgreSQL Service on Railway

1. Go to your Railway dashboard: https://railway.com/project/748fa91d-94c9-4a6e-af82-7b8250620c34?environmentId=f40f1d40-7fa8-4ee4-8f6f-1644a903be08
2. Click **"New" → "Database"** (or search for **PostgreSQL**)
3. Select the **PostgreSQL** template from Railway
4. Configure:
   - **Name:** `PostgreSQL` (or `oxot-db`)
   - **Tier:** Standard or higher (need pgvector extension)
   - **Volume:** 1GB (default, adjustable later)
   - **Region:** Same as OXOT-Website-JULY2026 (us-west2)

5. **Important:** Click the service → **Variables** → ensure these are set:
   ```
   POSTGRES_USER=oxot
   POSTGRES_PASSWORD=<auto-generated>
   POSTGRES_DB=oxot
   DATABASE_URL=postgresql://...  (auto-generated)
   ```

6. Once deployed, **note the DATABASE_URL** — it looks like:
   ```
   postgresql://oxot:<password>@<host>:<port>/oxot
   ```

### Step 3: Connect PostgreSQL to OXOT-Website Service

1. In the OXOT-Website-JULY2026 service → **Variables**
2. Add a **reference variable**:
   - Key: `DATABASE_URL`
   - Value: `${{ PostgreSQL.DATABASE_URL }}` (or whatever you named the PostgreSQL service)

3. This automatically pulls the connection string from PostgreSQL without hardcoding secrets.

### Step 4: Set Required Environment Variables

In the OXOT-Website-JULY2026 service → **Variables**, set:

| Variable | Value | Notes |
|----------|-------|-------|
| `AUTH_SECRET` | Generate a random 32+ char string | Session encryption. Use a password manager or `openssl rand -base64 32` |
| `SETTINGS_SECRET` | (Optional) Random 32+ char string | Falls back to AUTH_SECRET if unset |
| `OPENROUTER_API_KEY` | Your API key (optional) | For remote LLM. If empty, uses Ollama fallback |
| `NODE_ENV` | `production` | Already set in railway.json |
| `DEFAULT_LOCALE` | `en` | Default language |
| `SUPPORTED_LOCALES` | `nl,en` | Comma-separated locales |
| `LLM_PRIMARY` | `openrouter` | Use remote LLM (set to `ollama` for local) |
| `OPENROUTER_MODEL` | `openai/gpt-4o-mini` | Model to use for chat |
| `EMBED_DIM` | `2560` | Must match embedding model output dim |
| `OLLAMA_HOST` | `http://ollama:11434` | Local Ollama fallback |
| `OLLAMA_CHAT_MODEL` | `qwen3.5:9b` | Local chat model |
| `OLLAMA_EMBED_MODEL` | `qwen3-embedding:4b` | Local embedding model |

The `railway.json` in this repo already includes these variable definitions with sensible defaults.

### Step 5: Trigger a Full Deployment

1. **Merge PR #6** (Dockerfile fix) to main
2. Push any changes (if you made them locally):
   ```bash
   git add .
   git commit -m "infra: add railway.json variables for production deployment"
   git push origin main
   ```

3. Go to the OXOT-Website-JULY2026 service on Railway
4. Click **Deploy** → select **main** branch (should auto-trigger from the PR merge)

5. Wait for deployment phases:
   - **BUILD_IMAGE** — Builds the Docker image (should now succeed with PR #6)
   - **PRE_DEPLOY** — Runs migrations + seed scripts (requires DATABASE_URL to be set)
   - **DEPLOY** — Starts the app
   - **HEALTH_CHECK** — Hits `/en` endpoint

### Step 6: Verify Everything is Running

Once deployed:

1. Check the **OXOT-Website-JULY2026** service:
   - Status should be **"Running"**
   - Replica count: **1 running**

2. Check the service logs:

   # seeded en/cyber-digital-twin (... chars)

3. Visit the service domain (e.g., `https://oxot-website-july2026-production.up.railway.app`)
   - Should load the homepage in English
   - Admin panel should be accessible at `/admin`

## Infrastructure as Code

The configuration is entirely in the repo:

- **railway.json** — Main deployment config (build, deploy, variables)
- **docker-compose.yml** — Local dev setup (same postgres + pgvector as production)
- **.env.example** — Template for environment variables
- **scripts/migrate.mjs** — Database migration runner
- **scripts/seed-pages.mjs** — Import markdown content into pages table
- **scripts/seed-site-blocks.mjs** — Seed homepage blocks from i18n dictionaries
- **db/migrations/** — 15 SQL migration files (auto-run in preDeployCommand)

To replicate this deployment in another Railway environment:

1. Create a new PostgreSQL service (same version + pgvector)
2. Set the same variables (AUTH_SECRET, etc.)
3. Push the repo branch with this railway.json
4. Deploy — all configuration is in the code, no manual steps needed (except creating PostgreSQL)

## Environment Variables Reference

### Database
- **DATABASE_URL** — PostgreSQL connection string (set via reference variable)

### Authentication
- **AUTH_SECRET** — Session encryption secret (required, no default)
- **SETTINGS_SECRET** — Optional encryption for stored API keys

### Localization
- **DEFAULT_LOCALE** — Default language (default: `en`)
- **SUPPORTED_LOCALES** — Comma-separated locales (default: `nl,en`)

### LLM / AI
- **LLM_PRIMARY** — Primary LLM provider: `openrouter` or `ollama` (default: `openrouter`)
- **OPENROUTER_API_KEY** — API key for remote LLM (optional)
- **OPENROUTER_MODEL** — Model to use (default: `openai/gpt-4o-mini`)
- **OLLAMA_HOST** — Local Ollama endpoint (default: `http://ollama:11434`)
- **OLLAMA_CHAT_MODEL** — Local model for chat (default: `qwen3.5:9b`)
- **OLLAMA_EMBED_MODEL** — Local embedding model (default: `qwen3-embedding:4b`)
- **EMBED_DIM** — Embedding dimension (default: `2560`, must match model)

### Runtime
- **NODE_ENV** — Always `production` in Railway

## Database Schema

The app auto-initializes the schema on first deploy via `npm run db:migrate`. The migrations create:

- **content_chunks** — Embedded page content for RAG retrieval
- **visitor_sessions** — Consent-gated session tracking
- **visitor_events** — Behavioral analytics (page views, clicks, dwell time)
- **agent_messages** — LLM conversation history with citations
- **files** — File storage (images, documents)
- **pages** — CMS pages with frontmatter metadata
- **admin_users** — Admin accounts with encrypted passwords
- **site_blocks** — Configurable site sections (homepage hero, services, etc.)
- **ai_settings** — Admin-configurable LLM/embedding settings
- **menu** — Navigation menu structure
- **inquiries** — Contact form submissions

All tables include timestamps and proper indexes for performance.

## Content Seeding

On deploy, the app runs two seed scripts:

1. **npm run seed:pages** — Imports markdown files from `content/pages/<locale>/*.md` into the database
2. **npm run seed:site** — Loads homepage sections from `src/i18n/dictionaries/<locale>.json`

The scripts use `ON CONFLICT ... DO NOTHING` so they never clobber content the admin has edited in the UI. Safe to redeploy.

## Secrets Management

### Sensitive Variables
- **AUTH_SECRET** — Generate a random string. Railway stores it encrypted at rest.
- **SETTINGS_SECRET** — Same as above (optional).
- **OPENROUTER_API_KEY** — If you have an OpenRouter account, add your key here. If empty, the app falls back to Ollama.

### How They're Used
- **AUTH_SECRET** → Encrypts admin session cookies
- **SETTINGS_SECRET** → Encrypts the OpenRouter API key if the admin stores it via the UI (instead of via env var)
- **OPENROUTER_API_KEY** → Used directly in API calls for LLM inference

None of these are hardcoded or logged. Railway's Variables UI is encrypted at rest, and the app only reads them at startup.

## Troubleshooting

### Build Fails with "Module not found: Can't resolve '@/...'"
**Cause:** Dockerfile has `ENV NODE_ENV=production` before `npm ci`.
**Fix:** Merge PR #6.

### Deployment Fails at preDeployCommand
**Cause:** DATABASE_URL is not set or PostgreSQL is not running.
**Fix:**
1. Verify PostgreSQL service is deployed and running
2. In OXOT-Website-JULY2026 variables, check DATABASE_URL is set to `${{ PostgreSQL.DATABASE_URL }}`
3. Redeploy

### App Starts but Shows Errors
**Check logs:**
```
# Build logs
Railway Dashboard → OXOT-Website-JULY2026 → Deployments → (latest) → Build Logs

# Runtime logs
Railway Dashboard → OXOT-Website-JULY2026 → Deployments → (latest) → Runtime Logs
```

### "health check failed"
**Cause:** App crashed or didn't bind to port 3000.
**Check:**
- Runtime logs for errors
- DATABASE_URL is valid and reachable
- AUTH_SECRET is set (session errors will crash the app)

## Scaling & Maintenance

### Increase Database Size
1. Go to PostgreSQL service → Settings
2. Increase Volume size (pay as you grow)
3. No downtime for this operation

### Add More App Replicas
1. In railway.json, update `deploy.numReplicas` (or set via UI)
2. Push & redeploy
3. Railway load-balances traffic across replicas

### Backup Database
Railway automatically backs up PostgreSQL. To restore:
1. Go to PostgreSQL service → Settings → Backups
2. Restore to a point in time

## Updating Content

All content (pages, site blocks) is in the repo:

- **Pages:** `content/pages/<locale>/<slug>.md` (markdown with frontmatter)
- **Site Blocks:** `src/i18n/dictionaries/<locale>.json` (JSON)

Push changes → They're seeded on the next deploy (no clobbering if admin edited them).

For admin-created content (that doesn't go in the repo), use the Railway UI to manage it. The seed scripts skip conflicting records.
